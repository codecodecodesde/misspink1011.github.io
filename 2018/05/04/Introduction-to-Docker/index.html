Introduction to Docker

Posted on 2018-05-04

The ambitious goal of Docker is to package an application along with all of its dependencies easily and then run it smoothly in disparate development, test and production environments.

Docker solves the following problems:

  • conflicting dependencies: run in separate Docker container
  • missing dependencies: all dependencies have been packaged along with the application in a container.
  • platform differences: if both systems run DOcker, the smae container will execute without issues

Core Tech of Docker

Docker provides tools to make creating and working with containers as easy as possible.

Namespaces

Linux Containers and LXC, a user-space control package for Linux Containers, constitute the core of Docker. LXC users kernel-level namespaces to isolate the container from the host.

  • user namespace: separats the container’s and the host’s user databases, thus ensuring that the container’s root user does not have root privileges on the host.
  • process namespaces: only displays and manages processes running in the container, not the host.
  • network namespace: provides the container with its own network device and virtual IP address.

Control Groups

Control Groups implement resource accounting and limiting. It allows Docker to limit the resourcecs being consumed by a container, such as memory, disk space and I/O. And, it also outputs lots of resource metrics to monitor the resource consumption of the various processes within the containers and make sure that each gets only its fair share of the available resources.

Advanced Multi-Layered Unification Filesystem

AuFS is a layered filesystem that can transparently overlay one or more existing filesystems.

  • copy-on-write
  • use certain images as the basis: for example, use CentOS Linux image that can be used as the basis for many different containers.
  • version container images: each new version is simply a diff of changes from the previous version, effectively keeping files to a minimum.

Container vs. Virtual Machine

Container virtualize at the OS level, whereas hypervisor-based solution virtualize at the hardware level. While the effect is similar, the differences are important and significant.

  1. On the VM side, a hypervisor make siloed slices of hardware available. In contract, container make available protected portions of the operating system. Two containers running on the same OS, don’t know that they are sharing resources because each has it own abtracted network layers, processes and so on.
  2. VM provides access to hardware only, you still need to install an operating system. As a result, there are multiple full-fledged operating systems running, one in each VM, which quickly gobbles up resources on the server, such as RAM, CPU and bandwidth. Containers piggyback on an already running operating system as their host environment. They merely execute in spaces that are isolated from each other and from certain parts of the host OS. This has two significant benefits.
    • efficient resource utilization If a container is not executing anything, it is not using up resources, and containers can call upon their host OS to satisfy some or all of their dependencies.
    • cheap and fast to create and destroy There is no need to boot and shut down a whole OS. Instead, a container merely has to terminate the processes running in its isolated space, which is more akin to starting and quitting an application and is just as fast.

Docker Repositories

One of Docker’s killer features is the ability to find, download and start container images that were created by other developers quickly. The place where images are stored is called a registry, and Docker Inc. offers a public registry also called the Central Index. You can think of the registry along with the Docker client as the equivalent of Node’s NPM, Perl’s CPAN or Ruby’s RubyGems.

Docker in Action

Docker consists of a single binary that can be run in one of three different ways.

  • run as a dæmon to manage the containers: exposes a REST-based API that can be accessed locally or remotely. ( a lot of client libraries are available…)
  • issue instructions from the command line.
  • function as a client to remote repositories of images: pull or push images to the registry.
